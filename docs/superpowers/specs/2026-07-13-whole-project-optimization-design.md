# ZwZ 全项目代码优化设计

日期：2026-07-13

## 目标

在不改变任何对外可观察行为的前提下，对 `Sources` 下全部产品代码进行全面、深度优化，覆盖性能、内存、代码结构、稳定性、安全性与可维护性。当前未提交的批量重命名功能视为最新产品功能，必须保留并纳入优化与验证。

## 已确认约束

- 保持 GUI 操作与外观、CLI 命令与输出文本、`ZwzCore` 公共 API、配置数据及既有归档兼容性。
- 归档输出的所有可确定部分必须逐字节一致。随机数、nonce 等固有随机字段继续遵循现有协议、参数和分布。
- 不新增、替换或升级第三方依赖，保持当前依赖及版本策略。
- 不提高最低系统、Xcode 或 Swift 版本。没有降低 macOS 版本的硬性目标，仅在零风险且收益明确时考虑。
- 优化范围是 `Sources` 下的产品代码。测试可新增或调整用于验证，但测试、脚本和文档本身不作为优化对象。
- 允许深度重构、重写内部实现、拆分超大文件和删除经静态检查与测试证明无作用的死代码。
- 逐个审查产品文件并尽可能改善结构、命名、注释、重复逻辑和风格；不为制造改动而修改没有明确收益的文件。
- 可以直接修复证据充分的明显缺陷和安全问题；涉及正常行为或存在解释争议时先询问用户。
- 内部按阶段执行并持续验证，最终统一交付。
- 不创建 Git 提交，所有改动保留在工作区供用户检查。

## 选定策略

采用审计驱动的分层深度优化：先锁定兼容性与性能基线，再依次优化核心库、CLI 和 GUI，最后完成全项目整理与综合复核。该策略允许深度内部重构，同时通过分层验证缩小回归定位范围。

## 总体架构与边界

### ZwzCore

优先处理压缩、解压、预览、加密、分卷、文件 I/O、取消与错误恢复。沿写入链路“输入枚举 → 分块读取 → 压缩/加密 → 顺序写入 → 索引/分卷”和读取链路“格式识别 → 索引读取 → 验证/解密 → 解压 → 安全落盘”逐段测量与优化。

重点检查重复读取、无界复制、临时 `Data`、重复解析、锁竞争和不必要的串行等待。任何改变都必须保持块顺序、确定性二进制输出、进度语义、取消语义和错误语义。

### zwz CLI

整理参数解析、命令分发、公共工作流和资源清理。现有命令、参数、帮助及错误文本、输出顺序和退出状态作为固定契约，不允许改变。

### ZwzGUI

拆分职责过重的类型和超大文件，减少主线程工作、重复状态和无效刷新。文件操作、压缩、密码派生和密钥操作继续在后台执行，仅在主线程更新界面状态。布局、动画、文案、快捷键、弹窗顺序与持久化格式保持一致。

## 执行顺序

1. 建立保护网：锁定公共 API、CLI 输出、配置编码、归档黄金字节、错误类型和关键 GUI 状态；建立 Release 耗时与峰值内存基线。
2. 优化 `ZwzCore`：按读写数据链路测量和优化性能、内存、并发及资源生命周期。
3. 优化 CLI：统一内部执行骨架与验证逻辑，保持全部外部契约。
4. 优化 GUI：拆分职责、降低主线程负载和重复刷新，保持视觉与交互一致。
5. 全项目整理：逐文件处理重复逻辑、内部命名、注释、复杂条件、资源生命周期与死代码。
6. 综合复核：运行全量测试、Release 构建、标准性能基准、归档字节对比、安全回归和真实 macOS 手工验收。

关键路径若出现无合理解释的性能退化、不可确定的兼容风险，或以明显增加大文件内存耗尽风险换取微小速度提升，对应优化必须撤回。

## 错误处理、安全与并发

- 保留现有错误类型、错误优先级、提示文本、CLI 退出状态和 GUI 弹窗顺序。
- 临时文件、分卷中间产物与部分解压结果继续遵循既有清理及原子替换规则。
- 取消必须及时生效，且不能误删原文件、遗留半成品或允许迟到回调覆盖新状态。
- 路径穿越、符号链接逃逸、整数溢出、恶意长度、认证失败、签名无效与错误密钥必须在不安全写入前被阻断。
- 密码、私钥、派生密钥与解密上下文不得新增持久化、日志或不必要的生命周期。
- 并行化仅用于相互独立的计算或 I/O；输出顺序、进度回调与取消检查点必须兼容。
- 共享状态使用明确所有权与受控同步，重点审查 GUI 迟到回调、编辑会话、分卷写入和钥匙串操作竞态。
- 缺陷修复必须增加回归测试；涉及正常行为或存在争议的修复先取得用户确认。

## 性能基准

以 Release 构建为主，Debug 只要求测试及构建正常。使用固定生成规则创建代表性数据集，覆盖：

- 大量小文件；
- 单个大文件；
- 深层目录、Unicode 名称与隐藏文件；
- 高可压缩和不可压缩内容；
- ZIP、ZWZ V2、ZWZ V3；
- 单卷与分卷；
- 无加密、密码加密与公钥加密。

记录压缩、解压、列表、预览和单项提取的 Release 耗时与峰值内存。速度优先，允许为明显速度收益增加内存，但不得引入大文件内存耗尽风险。没有固定性能提升比例；关键路径不得无解释退化。

## 验证与完成标准

- 当前基线为 396 项测试执行、1 项因缺少桌面 RAR 样本而跳过、0 失败，总耗时约 544 秒。
- 优化后全部既有及新增测试通过，条件性跳过不得无故增加。
- Debug 测试与构建通过，Release 构建通过。
- 无随机归档输出执行完整哈希或逐字节比较。
- 含随机字段的归档验证固定字段、规范编码、协议参数、可读性和安全性质不变。
- 公共 API 编译契约、CLI 标准输出/错误输出/退出状态、配置序列化结果和 GUI 关键状态转换均有回归保护。
- 完成真实 macOS 手工验收清单，覆盖拖拽、文件关联、标签页、编辑、批量重命名、Touch ID/钥匙串、虚拟磁盘和打包 App；必须由用户操作的项目以用户反馈作为最终结果。
- 最终交付包含改动摘要、性能对比、验证证据、未改文件说明、已知限制和手工验收结果。

## 非目标

- 不增加新产品功能。
- 不重新设计界面或交互。
- 不改变公共 API、CLI 契约、配置格式或归档协议。
- 不优化测试代码、构建打包脚本或项目文档本身。
- 不调整第三方依赖。
- 不创建 Git 提交。
